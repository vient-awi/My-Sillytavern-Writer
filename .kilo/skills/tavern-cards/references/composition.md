# 创作执行

根据已确认的 `创作规划.yaml` 执行条目创作、质量检查、注册和错误恢复。

## 前置必读

开始创作前，必须先加载：

- `references/rules.md` — 正面写作规则（角色写作要点、世界观压缩）
- `references/conventions.md` — 条目注册约定（命名、文件格式、patch 命令）

**注意**：主代理只阅读正面规则。禁词清单（`references/rules-check.md`）由子代理在禁词扫描时使用，主代理不应在创作阶段阅读。

## entries 执行原则

- `创作规划.yaml` 的 `entries` 数组是项目级事实来源，按数组顺序执行。
- 必要时可因依赖关系调整执行顺序，但不得漏条目。
- 若调整会改变用户已确认的条目顺序或内容范围，先同步更新 `创作规划.yaml` 并询问用户确认。

### 编写顺序校验

**世界建构**：

1. 世界观概览先行——其他条目可能引用它，先确立整体框架
2. 区域条目按信息量决定是否拆分，逐区域展开
3. 历史事件最后写——依赖世界观和区域的设定

**角色**（对每个核心角色）：

1. 基础信息 — 必写
2. 性格调色盘 — 必写
3. 三面性 — 可选，仅当角色在不同场景下行为模式有根本性切换时（判断标准见 `references/requirements.md`）
4. 二次解释 — 必写

5 个以上角色时，先编写角色速览（`references/contents-creation/character/character-catalog.md`）。

**角色阶段原则**：

- 每个阶段的行为/变化倾向不是上一阶段的延续，而是该阶段独有的新模式
- 性格底色不会随阶段变化
- 衍生分为**通用衍生**（跨阶段始终存在）和**阶段专属衍生**（仅在该阶段出现）：专属衍生是该阶段才解锁的新行为，不是上一阶段衍生的升级版。同一行为在不同阶段的细微差异通过二次解释体现，不需要跨阶段复制衍生
- 其与三面性的差异在于：行为模式的变化不是由于场景的切换，而是随着剧情发展由人物关系的变化导致

## 条目类型与创作规则对应

| 条目类型 | 创作规则文档 |
|---------|------------|
| 世界观 | `references/contents-creation/worldbuilding/worldview.md` |
| 时间线 | `references/contents-creation/worldbuilding/timeline.md` |
| 地理 | `references/contents-creation/worldbuilding/geography.md` |
| 角色 | `references/contents-creation/character/basic-info.md` 等（见 character/ 目录） |
| NPC | `references/contents-creation/character/npc.md` |
| 事件 | 按 `references/rules.md` 编写，无专门文档 |
| 阶段指导 | `references/contents-creation/stage-guidance.md` |
| 扮演准则 | `references/contents-creation/presentation.md` |

### 时间线 条目

`时间线` 类型通过 `part` 区分：history（历史事件）、plot（当前剧情，仅从材料转化时编写）。

详见 `references/contents-creation/worldbuilding/timeline.md`。

### 阶段指导 条目

阶段化玩法/剧情主持策略条目，必须使用 EJS 条件渲染，通常集中到一条总指导条目。

详见 `references/contents-creation/stage-guidance.md`。

### 扮演准则 条目

呈现方式条目，世界书级别的整体条目（不是角色专属）。

**何时使用**：用户在以下情况需要呈现方式条目：
- 特定世界观下需要约束写作方式（如克苏鲁不希望血腥）
- 叙述风格需要统一指引（如"全篇比喻，但禁止重复意象"）
- 叙事基调需要明确（如"暗黑压抑，但留给希望"）

**rephrase**：用户有特定偏好时设为 true，如血腥规避等。

详见 `references/contents-creation/presentation.md`。

## 条目创作循环

对 `entries` 数组中的每个条目（按数组顺序）：

0. **规划完整度检查**：扫描 `创作规划.yaml` 中是否有 `# 待细化` 注释。如有，先追问用户补全后再创作。
1. **读取创作规则**：根据条目类型加载对应的创作规则文档（见上方对应表）
2. **编写条目内容**：根据规划信息 + 类型规则编写，遵循 `references/rules.md`
   - 从材料转化而来，且故事大纲或创作规划中存在 `important_chapters[].quotes[]` / `characters[].quotes[]` 原文引用字段时，优先把这些原句作为角色性格、行为边界、关系动态和世界观揭示的依据，避免只用转述概括。
3. **禁词扫描**（必须）：使用**独立检查上下文**扫描，有问题立刻修正。详见下方说明
4. **处理 EJS**（如需）：条目在规划文档的 `ejs.entries` 中时，按 `complexity` 一并处理（见 `references/conventions.md`）
5. **写入文件**：将条目内容写入对应文件路径
6. **立即注册**（不可跳过）：立即运行 patch 命令注册到 entryManifest（见 `references/conventions.md`）
   - 注册是断点续接的唯一判断依据，不注册 = 条目被视为未完成
   - 每个条目写完必须立即注册，禁止积攒多个条目后批量注册
7. 继续下一个条目

**约束**：步骤 3 不可跳过。每个条目写完后必须过禁词扫描才能注册——配置脚本无法检测内容质量，这是唯一的质量关卡。

注意：常见动作词不算八股，不要矫正过度。详见 `references/rules-check.md` 的例外说明。

### 独立检查上下文（禁词扫描）

禁词扫描应使用**独立检查上下文**执行，避免已有上下文干扰检查效果。调用 `check-agent`。

**检查内容**（按 `references/rules-check.md`）：
- 绝对零度 → 删除或改写
- 八股化 → 删除或改写
- 具体性 → 改写
- 翻译腔 → 改写
- 元叙事 → 删除
- 跨条目引用 → 补全或删除

检查不通过时修正内容，重新检查直到通过。

## 条目完成 DoubleCheck

所有条目编写完成后、进入技术层之前，必须执行以下检查。不通过则修正后重新检查。

### 完整性

- 规划文档中 `entries` 数组列出的条目是否都已编写
- 所有已编写的条目是否都已注册到 entryManifest（文件存在但未注册 = 未完成）
- 跳过的条目是否已记录原因

### 全局一致性

- 同一角色在不同条目中的描述是否矛盾（外貌、性格、关系）
- 世界观条目之间是否存在设定冲突
- 角色关系描述是否双向一致（A 条目说"与B是敌人"，B 条目也要对应）

### 全局禁词扫描

单条目扫描（独立检查上下文）聚焦于单个条目内部的问题。全局扫描关注跨条目问题：

- 上下文引入的八股化：单独看没问题，多个条目放一起重复了同一个比喻/句式
- 角色间标签化：A 和 B 都用了"温柔善良"，但两人的"温柔"应该完全不同
- 世界观压缩检查：按 `references/rules.md` 逐句检查

此阶段可使用独立检查上下文，将所有条目一起送检以发现跨条目问题。

### 格式规范

- YAML 格式正确（缩进、冒号、列表）
- XML 标签闭合正确
- 条目使用数据库格式，不是散文
- 命名和路径符合 `references/conventions.md`
- 所有条目内容无繁体字、日文汉字（检查角色名、地名、变量名等）
- entryManifest 中 keywords 无单汉字（如"樱""萤""玲"→改为"小樱""小萤""小玲"）

## MVU 一致性检查

MVU 和 EJS 编写完成后执行（SKILL.md 主流程第 7 步），详见 `references/mvu/guide.md` 收尾步骤第 4 步。

## 错误恢复

错误处理流程详见 `references/error-handling.md`。

### 常见场景速查

| 场景 | 处理方式 |
|------|----------|
| 信息不足（命名问题） | 归「关键信息确认」流程 |
| 信息不足（内容完整性） | 询问用户补充，或标记"未知" |
| 用户反馈不满意 | 更新创作规划.yaml → 修正 → 重跑禁词扫描 → 更新 abstract |
| 需求变更 | 评估影响 → 更新规划 → 同步修改条目 |
| patch 失败 | 解读错误 → 修正 → 重新执行 |
| 条目间一致性冲突 | 列出冲突 → 询问用户 → 更新规划 → 统一修改 |
