# MVU 模板文件修改规则

本文档说明复制到项目目录后的 MVU 模板文件作用、默认修改边界，以及模板修改的例外流程。仅在需要判断“某个模板能不能改、该怎么改”时查阅。

## 适用范围

本规则适用于从 `assets/mvu-templates/` 复制到项目目录的全部文件，以及配套注入的 MVU 运行时结构。

这些文件虽然位于项目目录中，但多数承载固定的运行时约定、脚本注册假设或格式契约。不了解完整链路时随意修改，通常只会引入错误。

## 默认规则

复制后的项目模板默认视为固定资产。

- **允许修改**：`状态栏界面.html`、`脚本/Zod.txt`
- **默认禁止修改**：除上述两项外的其余模板文件

如果用户没有明确要求修改模板文件，优先通过以下位置完成需求：

- `schema.ts`
- `initvar.yaml`
- `变量更新规则.yaml`
- EJS预处理 条目和 EJS 条件映射
- 普通世界书条目

不要把本应通过变量结构、变量初始值、变量更新规则或条目内容解决的问题，错误地转移到模板文件中处理。

## 各类模板的作用

### `状态栏界面.html`

它不是状态栏前端本体，只是加载外部状态栏页面的占位文件。默认内容是 GitHub CDN 链接占位符，用于在部署完成后替换为实际可访问的前端地址。

通常只允许修改其中的加载地址，例如将占位符替换为实际的 GitHub + jsDelivr 地址或其它已部署的可访问 URL。状态栏界面本身的开发、样式和交互修改应在独立前端项目中完成，见 `references/packaging.md`。

### `脚本/Zod.txt`

运行时使用的 Zod Schema。它不是独立设计文件，而是 `schema.ts` 的内联副本。允许修改的本质只有一种：将最新的 `schema.ts` 同步进去。

除同步 `schema.ts` 外，不要单独在 `脚本/Zod.txt` 中添加、删除或重写 Schema 规则。

### 其它模板文件

包括变量列表、变量输出格式、变量更新中美化、变量更新美化、MVU 脚本及其它配套模板。这些文件通常服务于固定的数据流、脚本注册或提示词格式约定，默认不需要修改。

#### `变量列表.txt`

其中的 `{{format_message_variable::stat_data}}` 是酒馆助手宏，用于向酒馆 AI 展示 YAML 格式的 MVU 变量。可以通过修改 `::` 之后的变量地址，改变展示给 AI 的变量内容。

除非用户明确要求调整 AI 可见变量范围，或已确认问题根源在展示变量地址，否则不要修改这个宏。

#### `变量输出格式.txt`

用于告诉酒馆 AI 如何更新 MVU 变量。通常不需要修改。

#### `变量更新中美化.html` 与 `变量更新美化.html`

用于向用户展示美化后的变量更新界面。通常不需要修改。

#### `MVU.txt`

引入 MVU 脚本，严禁修改。

AI 若在不了解完整上下游约束的情况下修改这些模板，极易出现：

- 变量展示格式与脚本解析不匹配
- 正则脚本触发条件失效
- 运行时依赖字段缺失或命名漂移
- 表面可读但实际不可执行的提示词结构

因此，除非进入例外流程，否则一律保持原样。

## 模板修改例外流程

只有满足以下任一条件时，才允许修改模板文件：

- 用户明确要求修改某个模板文件
- 已确认问题根源就在模板本身，且不能通过 `schema.ts`、`initvar.yaml`、`变量更新规则.yaml`、EJS 映射或普通条目修复

进入例外流程时，按以下顺序处理：

1. 先说明为什么不能在非模板层解决
2. 明确指出要修改的具体模板文件
3. 说明该文件承担的作用，以及本次修改会影响哪一段链路
4. 完成修改后执行对应检查

若无法清楚说明以上三点，则不应修改模板。

## 修改后检查

### 修改 `脚本/Zod.txt`

必须检查：

- 内容是否与最新 `schema.ts` 一致
- 是否仅为内联同步，而非额外手写规则
- `export type` 等 SillyTavern 不支持的 TypeScript 语句是否已移除

### 修改 `状态栏界面.html`

必须检查：

- 是否仅修改了外部加载地址或占位符
- 目标 URL 是否指向已部署的状态栏前端页面
- 路径是否与部署产物一致，例如 `dist/{ProjectName}/界面/状态栏/index.html`
- 若未部署状态栏前端，则保持占位符原样，不要臆造链接或内联编写界面逻辑

### 例外修改其它模板

视为高风险操作，必须额外检查：

- 改动是否破坏既有脚本注册或调用约定
- 改动后的格式是否仍被下游脚本、正则或运行时正确消费
- 是否需要同步更新相关文档或其它依赖文件

如果以上任一项无法确认，停止修改并回到非模板方案。

## 决策顺序

当需求看起来“可能要改模板”时，按以下顺序决策：

1. 先看是否能通过 `schema.ts` 调整变量结构
2. 再看是否能通过 `initvar.yaml`、`变量更新规则.yaml` 或 EJS 映射解决
3. 再看是否只是普通世界书条目内容不足
4. 只有前三项都不能解决，才考虑模板例外修改

核心原则：**默认不改模板；真要改，也只改知道自己为什么要改的那一层。**
