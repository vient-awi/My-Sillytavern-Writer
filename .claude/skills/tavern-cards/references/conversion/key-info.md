# 关键信息确认

在生成条目规划前，必须完成关键信息确认。

## 识别标准

以下信息必须明确，不得使用占位符：

**地名类**：
- 城市名、区域名、街道名
- 学校名、公司名、组织名
- 特定场所名（酒吧、医院、公园等）

**角色名类**：
- 所有有台词或重要戏份的角色全名
- 角色的昵称、代号、别名
- 组织名称及其成员

**专有名词类**：
- 特殊能力、技能名称
- 特定物品、道具名称
- 世界观专有术语（如修炼等级、种族名称等）

**时间类**：
- 具体年份、日期（如故事发生在2023年）
- 特定时间点（如"三年前的那场事故"）

## 问题列表格式

```yaml
关键信息确认清单:
  地名类:
    - 问题: "源材料中提到的'那所学校'具体叫什么名字？"
      上下文: "第3章，苏云提到自己是这所学校的学生"
      建议: "可以是虚构名称，如'星辰学院'，或使用代号如'第一高中'"
    
    - 问题: "故事发生的主城市叫什么？"
      上下文: "全文多处提及'这个城市'但未给出具体名称"
      建议: "建议使用有特色的名字，如'雾城'、'星海市'"
  
  角色名类:
    - 问题: "第7章出现的'那个黑衣人'是否有名字？"
      上下文: "在第7章和第12章都有出场，是重要反派"
      建议: "如原文未提及，可询问用户是否需要命名"
  
  专有名词类:
    - 问题: "源材料中的'那种力量'具体叫什么？"
      上下文: "第5章首次出现，是核心设定"
      建议: "需要一个正式名称，如'灵能'、'源力'等"
```

## 确认流程

1. **收集问题**：通读源材料时，遇到关键信息缺失立即记录
2. **整理清单**：按类别整理问题列表，提供上下文和建议
3. **一次性确认**：在生成条目规划前，将完整问题列表提交用户确认
4. **记录答案**：将用户确认的答案记录到 `故事大纲.yaml` 的 `confirmed_info` 段落
5. **验证完整性**：确认所有关键信息都已明确后，方可进入下一阶段

## 确认原则

- **一次性原则**：所有问题整理完毕后一次性提交，避免反复打扰用户
- **上下文原则**：每个问题必须提供原文上下文，帮助用户理解问题来源
- **建议原则**：对每个问题提供建议答案，降低用户思考负担
- **不捏造原则**：用户未确认的信息不得自行补全，必须暂停等待确认

## 边界说明

**关键信息确认只处理命名问题**：
- 地名、人名、组织名、术语名等具体名称
- 源材料中用代词、模糊词指代的具体事物
- 不涉及信息的完整性、详细程度

**不在此流程处理的问题**：
- 角色背景不够详细 → 归「信息缺失处理」（见 `references/error-handling.md`）
- 世界观设定有缺失 → 归「信息缺失处理」
- 角色关系描述不完整 → 归「信息缺失处理」
