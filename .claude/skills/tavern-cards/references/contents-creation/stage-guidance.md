# 阶段指导

阶段化玩法/剧情主持策略条目，用于跑团、无限流、副本、经营循环、任务制玩法等有固定流程的角色卡。

## 使用条件

`阶段指导` 必须同时满足：

- 项目使用 MVU 记录当前阶段
- 项目使用 EJS 根据阶段变量控制段落渲染
- 条目通常合并为一个总指导，而不是每个阶段拆一个条目

如果不需要 EJS 控制，不使用 `阶段指导` 类型，改写为普通 `扮演准则` 或 `事件`。

## 与其他类型的边界

- 不是时间线：时间线写事件顺序，阶段指导写每个阶段怎么主持。
- 不是事件：事件写单次触发内容，阶段指导写可复用流程规则。
- 不是扮演准则：扮演准则写全局呈现方式，阶段指导写阶段化玩法推进。

## 内容结构

每个阶段只写当前阶段需要影响 AI 输出的主持规则：

```yaml
<%_ if (phase === 'explore') { _%>
explore_phase:
  goal: 自由探索并发现异常线索
  narration_focus:
    - 强调环境细节和可调查对象
    - 用异常痕迹制造下一步行动方向
  interaction:
    - 允许<user>主动选择调查目标
    - NPC 只提供局部信息，不直接给出答案
  progression_boundary:
    - 未满足关键线索前，不进入战斗阶段
<%_ } else if (phase === 'battle') { _%>
battle_phase:
  goal: 处理正面冲突
  narration_focus:
    - 明确敌我位置、威胁来源和行动后果
    - 每轮给出可响应的战术压力
  interaction:
    - 根据<user>行动推演战局变化
    - 不替<user>决定攻击、防御或逃跑
  progression_boundary:
    - 冲突解除后才进入结算阶段
<%_ } _%>
```

## 编写要点

- 阶段名称必须与 MVU 阶段变量的取值一致。
- 每个阶段写主持目标、叙事重点、互动方式、推进边界。
- 推进边界写“什么时候不能进入下一阶段”，比只写“什么时候进入下一阶段”更可靠。
- 不写具体剧情结果，结果应由 <user> 行动和变量状态推演。
- 不在阶段指导里重复世界观、角色设定、事件全文。
- 不替 <user> 决定行动、台词、心理。

## EJS 要求

- 条目必须使用条件渲染，见 `references/ejs/guide.md` 的“段落控制（条件渲染）”。
- 条件中引用的阶段变量必须在 `EJS预处理` 条目中通过 `define()` 注册，该条目使用 `@@generate_before` 装饰器。
- state 中该条目的 `ejs.mode` 为 `conditional`，`condition` 为空字符串。

## 常见错误

- 把阶段指导写成剧情大纲：只规定固定剧情，不给主持策略。
- 把阶段指导写成事件：每个阶段只写一次性触发桥段。
- 把所有阶段同时渲染：没有用 EJS 根据当前阶段裁切。
- 阶段边界过松：AI 提前跳到高潮、结算或下一副本。
- 阶段边界过死：无论 <user> 做什么都不允许推进。

- [ ] 阶段变量已在 EJS预处理 中 define()，条目使用 @@generate_before
- [ ] 每个阶段都有 goal / narration_focus / interaction / progression_boundary
- [ ] 没有写成固定剧情大纲或单次事件
- [ ] 没有替 <user> 决定行动、台词、心理
