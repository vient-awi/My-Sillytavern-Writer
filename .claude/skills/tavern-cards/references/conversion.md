# 从材料转化

当用户有现成素材（轻小说、游戏脚本、小说等）时，需求对齐阶段的世界信息和角色信息不通过提问收集，改为从源材料提取。创作流程与从零创作完全相同。

## 流程

### 1. 评估材料类型

通读所有源材料后，根据内容特征判断结构类型，选择对应路径。详见 `references/conversion/assessment.md`。

- **结构化材料** → 跳过“叙事材料路径”，直接进入“生成条目规划”
- **叙事材料** → 按“叙事材料路径”：初读构建大纲 → 整理深化
- **混合材料** → 分别处理，以结构化部分为准，叙事部分补充

### 2. 叙事材料路径

信息散落在叙述中时，用大纲提取信息，确保提取不遗漏。详见 `references/conversion/outline.md`。

**初读构建大纲流程**：
```
确定章节规律 → grep 确定边界 → 判断分卷 → 判断文本长度
                                    ├─ 短文本（<3万字）→ 主代理直接处理
                                    └─ 长文本（≥3万字）→ 分配子代理任务 → 合并整理
```

初读完成后，进入整理深化：补充 `summary`、`story_summary` 等汇总信息，合并重复内容，整理原文引用，确保一致性。详见 `references/conversion/outline.md` 第七步。

长文本使用子代理串行处理，调用 `conversion-agent`，详见 `references/conversion/outline.md`。

**子代理职责**：
- 通读指定范围的内容
- 为每章撰写 notes（章节概要）
- 提取世界观、势力、角色、物品等关键信息
- 提取人物塑造、剧情高潮、核心世界观揭示的原文引用
- 收集缺失的具体名称（地名、人名、组织名、术语名），输出到 `missing_names` 字段
- 输出后运行 `scripts/validate-conversion-outline.mjs` 验证片段，验证不通过时修正后重跑

### 3. 关键信息确认

大纲构建完成后、生成条目规划前，必须完成关键信息确认。详见 `references/conversion/key-info.md`。

- 识别缺失的具体名称（地名、人名、组织名、术语名）
- 整理问题列表（合并子代理收集的缺失名称）
- 一次性提交用户确认
- 记录答案，验证完整性

### 4. 转化质量检查

大纲构建完成后、生成条目规划前，必须执行质量检查：

1. **格式、结构与引用验证**：运行 `node scripts/validate-conversion-outline.mjs 故事大纲.yaml 源文件.txt`，详细说明见 `references/conversion/validation.md`
   - 长文本分片由 `conversion-agent` 处理时，子代理应在输出片段后立即运行该脚本验证自己的片段。
   - 主代理合并整理出最终大纲后，仍需对最终大纲再运行一次脚本验证。
   - 脚本检查 YAML 语法、缩进、重复 key、章节引用一致性、占位符和原文引用真实性。

2. **大纲完整性检查**：
   - [ ] 所有章节都有 `chapters[].notes`，且内容具体（50-100 字）
   - [ ] 源材料中的关键世界观设定已提取
   - [ ] 主要角色、重要配角、关键关系都已记录
   - [ ] 角色的 `identity`、`personality`、`relationship` 各自独立
   - [ ] 角色的 `appearance` 只包含偏离默认认知的特征，无万能美人描写
   - [ ] 源材料明确提到的区域和场景已记录到 `regions[]`
   - [ ] 势力的 `territory` 和 `key_members`（源材料有提及时）已记录
   - [ ] 重要物品、能力、组织、地名、术语已记录
   - [ ] `important_chapters` 覆盖关键转折、首次出现、名场面、核心设定揭示
   - [ ] `style_hints` 已从源材料叙事方式推断填写
   - [ ] `missing_names` 中的问题已完成关键信息确认，确认结果已写回大纲或创作规划

3. **引用整理检查**：
   - [ ] `characters[].quotes[]` 已合并去重，保留能体现性格底色、行为边界、关系动态、说话方式的代表性原句
   - [ ] `important_chapters[].quotes[]` 每章最多 1-2 条，且能支撑章节重要性
   - [ ] 所有 `quotes[].text` 都未被改写、截断或拼接
   - [ ] 所有 `quotes[].chapter` 和 `important_chapters[].chapter` 都与 `chapters[].name` 完全一致
   - [ ] 引用不是普通信息句，确实有助于后续条目创作

4. **一致性检查**：
   - [ ] 同一角色在不同片段中的身份、性格、关系描述不矛盾
   - [ ] 同一设定、组织、物品在不同片段中的名称和功能不矛盾
   - [ ] 时间线、关系变化、物品获得/失去没有明显断裂
   - [ ] 子代理分片合并后没有重复章节或重复全局字段

5. **语义复核**：
   - [ ] 原文引用对应正确角色、正确章节
   - [ ] `context` 没有过度解释，确实由引用本身支撑
   - [ ] 大纲中的概括没有把原文行为弱化成抽象标签
   - [ ] 没有把原文没说的信息补成事实

步骤 1 由脚本执行，步骤 2-5 由主代理执行。检查不通过时修正内容，重新检查直到通过。

### 5. 生成条目规划

根据结构化材料或故事大纲，生成条目规划（写入编写规划文档，见 `references/requirements.md`）。

**叙事材料路径**：每个条目额外标注 `source_chapters`，用于精准复读。详见 `references/conversion/source-chapters.md`。

### 6. 确定 plot 条目组织形式

条目规划中包含 plot 条目时，询问用户按何种形式组织：

- **按章节分割**：每章一条独立条目，适合剧情按章节推进、章节间跳跃不大的情况
- **按时间分割**：每个时间段一条独立条目，适合多线叙事、时间跳跃频繁的情况

确定后在条目规划中标注，具体约束和 EJS 条件写法见 `references/contents-creation/worldbuilding/timeline.md`。

### 7. 进入创作流程

用户确认条目规划后，进入与从零创作相同的流程（见 SKILL.md 完整项目流程）。

编写每个条目时，结构化材料无需复读，直接按创作规则编写。叙事材料先根据 source_chapters 精准复读对应章节，再按创作规则编写。

---

## 转化特有约束

- **去重**:多个章节描述同一角色 → 合并为一个条目
- **不编造**:原文没说的不自行补全
- **文学→数据**:原文散文转为 YAML 数据库格式
- **信任距离**:每个提取特征必须有原文证据

---

## 错误处理

转化阶段的错误处理流程详见 `references/error-handling.md`:

- **材料矛盾处理**:识别矛盾 → 记录 → 询问用户 → 按确认修改 → 验证
- **信息缺失处理**:识别缺失 → 记录 → 询问用户 → 选择处理方式
- **转化失败恢复**:评估原因 → 选择策略 → 执行恢复
